# プロジェクト定義書（カジュアル版）

> 音楽ゲームのプレイ動画から難易度を推定し、「なぜ難しいのか」をAIで可視化する研究プロジェクト

---

## 1. 何を作るか

| 項目 | 内容 |
|------|------|
| **入力** | 音楽ゲームのプレイ動画（YouTube等から収集） |
| **出力① 推定** | 動画から推定したユーザー体感難易度（数値） |
| **出力② 可視化** | 「どのノーツ配置が難しさに効いているか」のヒートマップ |
| **正解ラベル** | 有志Wikiに掲載されているユーザー難易度（譜面定数） |

**一言で言うと：**  
公式難易度とユーザーの体感がズレる「詐称・逆詐称」という文化現象を、動画解析＋XAIで客観的に分析するシステム。

---

## 2. パイプライン全体像

```
YouTube動画
    ↓ [collection/downloader.py]
生動画ファイル (data/raw/)
    ↓ [preprocessing/extractor.py]
フレーム列
    ↓ [preprocessing/diff.py → binarize.py]
差分フレーム列（ノーツの動きだけ抽出）
    ↓ [preprocessing/dataset.py]
クリップ単位のテンソル（PyTorch Dataset）
    ↓ [models/cnn_lstm.py]
難易度スコア（順序回帰で推定）
    ↓ [xai/gradcam.py → visualize.py]
Grad-CAMヒートマップ（難所の可視化）
```

Wiki難易度ラベル → [collection/scraper.py] → data/labels/ に保存して学習時に参照

---

## 3. モジュール定義

### 3-1. collection（データ収集）

| ファイル | やること |
|----------|---------|
| `downloader.py` | yt-dlp を叩いて YouTube からプレイ動画を落とす |
| `scraper.py` | 有志Wiki をスクレイピングしてユーザー難易度ラベルを取ってくる |

**注意点：** 収集した動画・ラベルの対応づけ（曲名・難易度クラス）が後工程のキモなので、命名規則やCSVでの管理方法を早めに決める。

---

### 3-2. preprocessing（前処理）

| ファイル | やること |
|----------|---------|
| `extractor.py` | 動画からフレームを一定fps で抽出（OpenCV） |
| `diff.py` | 連続フレームの差分をとってノーツの動きだけ残す |
| `binarize.py` | 差分画像を二値化してノイズを除去 |
| `dataset.py` | 処理済みフレームを一定長のクリップにまとめてPyTorchのDatasetに仕立てる |

**予備実験の教訓：** 静止画1枚で分類すると休憩地帯（ノーツが少ない瞬間）で過学習する。  
→ クリップ単位（時系列）で扱うことでこれを回避する。

---

### 3-3. models（モデル）

| ファイル | やること |
|----------|---------|
| `cnn.py` | 各フレームから空間的な特徴を抽出（torchvisionのバックボーンを使用予定） |
| `lstm.py` | CNNの出力系列を時系列として処理（ノーツの出現パターンを学習） |
| `cnn_lstm.py` | CNN＋LSTMを結合し、末尾に順序回帰ヘッドを接続した統合モデル |

**順序回帰を使う理由：**  
難易度は「12 は 11 より難しい」という順序関係があるので、単純な多クラス分類よりも順序を考慮した回帰の方がRMSEが小さくなる（Franks et al. 2023）。

---

### 3-4. train（学習）

| ファイル | やること |
|----------|---------|
| `loss.py` | 順序回帰に対応した損失関数 |
| `trainer.py` | 学習ループ、バリデーション、チェックポイント保存 |
| `evaluate.py` | RMSE・順序一致度などの評価指標を計算 |

---

### 3-5. xai（説明可能AI）

| ファイル | やること |
|----------|---------|
| `gradcam.py` | CNNの勾配情報からどの領域に注目したかを計算（Grad-CAM） |
| `visualize.py` | ヒートマップを元フレームに重畳して画像・動画として保存 |

**目的：** 「AIが難しいと判断した場所」と「人間が難所と感じる場所」が一致するかを確認し、モデルの妥当性を担保する。

---

### 3-6. utils（共通ユーティリティ）

| ファイル | やること |
|----------|---------|
| `config.py` | `configs/config.yaml` を読み込んで各モジュールに渡す |
| `logger.py` | 学習ログ・エラーログの設定 |

---

## 4. データの置き場所

```
data/
├── raw/          生動画ファイル（mp4等）
├── processed/    前処理済みフレームクリップ
└── labels/       ユーザー難易度ラベル（CSV等）

outputs/
├── models/       学習済みモデル重み（.pth）
├── logs/         学習ログ
└── visualizations/  Grad-CAMヒートマップ画像
```

> `data/` と `outputs/` は .gitignore で除外する（容量が大きいため）

## 5. Docker構成

| サービス | 用途 | GPU |
|----------|------|-----|
| `train` | モデル学習・評価 | 使う |
| `collect` | 動画ダウンロード・スクレイピング | 不要 |
| `notebook` | 探索的実験（JupyterLab） | 使う |

ベースイメージ：`pytorch/pytorch:2.2.0-cuda12.1-cudnn8-runtime`

## 6. 技術スタック早見表

| 技術 | 使う場面 |
|------|---------|
| PyTorch + CUDA | モデル学習全般 |
| torchvision | CNNバックボーン（事前学習済みモデル） |
| OpenCV | フレーム抽出・差分・二値化 |
| yt-dlp | YouTube動画ダウンロード |
| requests + BeautifulSoup4 | Wikiスクレイピング |
| NumPy / Pandas | データ整形・ラベル管理 |
| scikit-learn | 評価指標（RMSE等） |
| matplotlib | Grad-CAMヒートマップ描画 |
| JupyterLab | 実験ノートブック |
| PyYAML | config.yaml 読み込み |
| tqdm | 進捗バー |
